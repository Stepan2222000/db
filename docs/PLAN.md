# План реализации: CLI `db` для управления PostgreSQL-стеком

## Контекст

Спецификация и flow-документ готовы:
- `docs/SPEC.md` — полная спецификация
- `docs/SPEC.flow.yaml` — flow-граф с definitions и modules

Система реализуется на Python. Принцип DRY: три слоя, каждый следующий использует предыдущий. Ни одна команда не дублирует логику другой — всё через общие примитивы и операции. Реализация инкрементальная — 10 этапов, каждый тестируется и коммитится отдельно.

## Архитектура (ориентировочная, может меняться при реализации)

Три слоя:

**Слой 1 — core (примитивы).** Низкоуровневые модули, которые ничего не знают о бизнес-логике. Чтение и запись .env файлов. Генерация compose.yaml из набора конфигов. Subprocess-обёртки для docker. Paramiko-обёртки для SSH/SFTP. Цветной вывод и логирование.

**Слой 2 — operations (переиспользуемые блоки).** Бизнес-логика, собранная из примитивов core. Обнаружение сервисов через glob. Ожидание pg_isready. Стрим pg_dump. SQL-запросы через docker exec. Синхронизация паролей. Алгоритм ротации. Управление cron. Сбор метрик.

**Слой 3 — commands (CLI).** Тонкие обёртки, которые связывают пользовательский ввод с операциями. Каждая команда — отдельный модуль. Typer как фреймворк.

Структура каталогов:
```
db/
  .env                    # глобальный конфиг
  compose.yaml            # генерируется автоматически
  metrics.db              # SQLite метрики
  .gitignore
  services/               # сервисные .env файлы
  data/                   # PostgreSQL datadir по сервисам
  dumps/                  # локальные дампы
  ops/                    # Python-код
    cli.py                # точка входа
    requirements.txt
    core/                 # слой 1
    operations/           # слой 2
    commands/             # слой 3
```

## Этапы реализации

### Этап 1: Фундамент — парсер конфигов, обнаружение сервисов, генерация compose

**Цель:** построить базовый слой, на котором держатся все остальные этапы.

**Делаем:**

Парсер .env файлов:
- чтение KEY=VALUE из файла в dict
- запись dict обратно в файл
- поддержка комментариев и пустых строк
- фиксированные ключи сервиса: `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_PORT`
- опциональные ключи сервиса: `POSTGRES_MAX_CONNECTIONS`, `POSTGRES_BACKUP_DISABLED`, `POSTGRES_BACKUP_FORMAT`, `POSTGRES_MEMORY_LIMIT`, `POSTGRES_CPU_LIMIT`
- используется и для глобального `.env`, и для сервисных `services/.env.*`

Обнаружение сервисов:
- glob по `services/.env.*`
- из каждого имени файла извлечь имя сервиса
- вернуть список имён
- это единственный способ узнать какие сервисы существуют

Генерация compose.yaml:
- прочитать глобальный `.env` и все сервисные конфиги
- собрать Docker Compose структуру в Python dict
- для каждого сервиса сгенерировать блок:
  - образ `postgres:18`
  - `container_name` = имя сервиса
  - `env_file` = `services/.env.<service>` с `format: raw`
  - derived `environment.POSTGRES_DB=<service>`
  - проброс порта на `5432`
  - volume с учётом PGDATA для PostgreSQL 18
  - `restart: unless-stopped`
  - healthcheck: interval 10s, timeout 5s, retries 3, start_period 30s
  - `shm_size` = RAM * 0.9 / количество сервисов (из `/proc/meminfo`)
  - `command: postgres -c max_connections=<N>` если задано
  - `deploy.resources.limits` если заданы memory/cpu
- первая строка файла — комментарий «не редактировать руками»
- записать через `yaml.safe_dump`

Хелперы вывода:
- ANSI-цвета для [OK], [ERROR], [WARN]
- print helpers

**Не делаем:** docker-команды, CLI, SSH, бэкапы, метрики. Ничего не запускается — только генерируются файлы.

**Результат:** вручную создать `.env` и `services/.env.test`, запустить генерацию — получить валидный compose.yaml.

**Flow:** `[[global-env]]`, `[[service-env-file]]`, `[[generated-compose]]`

### Этап 2: Docker-обёртка и каркас CLI

**Цель:** научить систему общаться с Docker и дать оператору точку входа.

**Делаем:**

Subprocess-обёртки для docker:
- `compose up -d` (один сервис или весь стек)
- `compose stop` (один сервис)
- `compose down`
- `exec` с поддержкой pipe stdout (для pg_dump)
- `stats --no-stream` (для метрик)
- каждая обёртка возвращает stdout/stderr
- проверка exit code, при ошибке — полный traceback

Typer CLI:
- точка входа `cli.py`
- 8 подкоманд-заглушек: add, remove, apply, backup, dump, restore, sizes, autobackup
- каждая пока выводит «не реализовано»

Зависимости:
- `requirements.txt`: typer, pyyaml, paramiko, python-crontab

**Не делаем:** логику команд, SSH, бэкапы, метрики. Только инфраструктура.

**Результат:** `python ops/cli.py --help` показывает все 8 команд.

**Flow:** `[[runtime-tooling]]`

### Этап 3: Команды `db add` и `db apply`

**Цель:** первые рабочие команды — можно создать сервис и применить изменения конфига.

**Делаем:**

`db add <name> [--port X] [--password Y]`:
- проверить имя: только буквы, цифры, подчёркивания
- проверить что `services/.env.<name>` не существует
- если порт не указан — подобрать свободный начиная с 5401
- если указан — проверить диапазон 1024..65535 и уникальность
- создать `services/.env.<service>` (user=admin, пароль из аргумента или дефолт из .env, порт)
- создать `data/<service>/`
- перегенерировать compose.yaml
- показать итоговые параметры, спросить подтверждение
- `docker compose up <service> -d`
- ждать `pg_isready` до 60 секунд
- вывести параметры подключения
- если что-то упало — traceback, без отката

`db apply`:
- для каждого `services/.env.*` без каталога data/ — создать каталог
- перегенерировать compose.yaml (включая пересчёт shm_size)
- проверить конфликты портов
- показать diff, спросить подтверждение
- `docker compose up -d`
- для каждого running-контейнера — ALTER ROLE с паролем из .env

Новые операции (переиспользуются дальше):
- pg_isready: цикл проверки готовности, до 60 секунд
- pg_query: выполнение SQL через `docker exec ... psql`
- password_sync: ALTER ROLE для каждого running-контейнера, идемпотентно

**Не делаем:** удаление, бэкапы, SSH, метрики, sizes.

**Результат:** `db add mytest` создаёт сервис, `psql` подключается. Правка пароля + `db apply` — пароль обновляется.

**Flow:** модули `create-service`, `sync-stack`, definition `apply-password-sync`

### Этап 4: Команда `db sizes`

**Цель:** оператор видит размер каждой базы.

**Делаем:**

`db sizes`:
- пройти по всем сервисам через service discovery
- для running: запросить `pg_database_size()` через pg_query
- для stopped: сообщить что не запущен, предложить запустить
- если сервисов нет — сообщить явно
- размер в читаемом формате (МБ / ГБ)

**Не делаем:** удаление, бэкапы, SSH, метрики.

**Результат:** `db sizes` после создания сервиса показывает имя и размер.

**Flow:** модуль `inspect-sizes`

---

### Этап 5: Команда `db remove`

**Цель:** замкнуть жизненный цикл — создание и удаление. Полный CRUD.

**Делаем:**

`db remove <name> [--force]`:
- проверить что `services/.env.<name>` существует
- если нет — показать список сервисов, выйти
- показать что будет удалено:
  - конфиг
  - контейнер и его статус
  - каталог данных и размер
- запросить подтверждения:
  - согласие y/n
  - повторный ввод имени сервиса
  - ввод слова DELETE
- `--force` пропускает все подтверждения
- остановить и удалить контейнер
- удалить `services/.env.<service>`
- удалить `data/<service>/`
- перегенерировать compose.yaml

**Не делаем:** удаление бэкапов на remote, SSH, метрики.

**Результат:** `db add temp && db remove temp --force` — создано и удалено.

**Flow:** модуль `remove-service`

---

### Этап 6: SSH/SFTP-обёртка (paramiko)

**Цель:** инфраструктура для работы с remote-сервером.

**Делаем:**

Paramiko wrapper:
- подключение по паролю (IP, логин, пароль из `.env`)
- открытие SFTP-сессии
- отправка файлового потока (для стриминга pg_dump напрямую)
- листинг каталога с датами и размерами
- удаление файлов
- переименование (`.tmp` → финальное имя)
- создание каталогов на remote
- отключение
- одно подключение переиспользуется для всех операций

**Не делаем:** логику бэкапов, ротацию, restore. Только transport.

**Результат:** подключиться к remote, создать файл, переименовать, получить список, удалить.

**Flow:** `[[runtime-tooling]]`, `[[remote-backup-storage]]`

### Этап 7: Команда `db dump`

**Цель:** ручной дамп локально. Операция pg_dump переиспользуется потом в `db backup`.

**Делаем:**

`db dump <name>`:
- найти запущенный контейнер
- если не запущен — сообщить, стоп
- запросить `pg_database_size()`
- проверить свободное место:
  - gzip: нужно размер_базы * 0.3
  - plain: нужно размер_базы * 1.0
  - если не хватает — показать сколько нужно / сколько есть, стоп
- стримить дамп: `docker exec pg_dump | gzip > dumps/<service>_YYYY-MM-DD_HH-MM.sql.gz`
- вывести путь и размер файла

Новая операция (переиспользуется в backup):
- pg_dump: subprocess.Popen, pipe stdout из docker exec, проверка exit code

**Не делаем:** отправку на remote, ротацию, SSH.

**Результат:** `db dump mytest` — файл в `dumps/`, размер > 0.

**Flow:** модуль `local-dump`

---

### Этап 8: Команда `db backup` — стрим на remote

**Цель:** автоматический бэкап — дамп стримится на remote без локального файла. Самая сложная команда.

**Делаем:**

`db backup [name]`:
- поставить lock (один процесс одновременно)
- подключиться к remote через SSH-обёртку (этап 6)
- если remote недоступен — ошибка в лог, стоп
- определить сервисы-кандидаты (все `services/.env.*` или один конкретный)
- пропустить сервисы с отключённым бэкапом
- для каждого сервиса:
  - `pg_dump | gzip` → SFTP → `<service>_YYYY-MM-DD_HH-MM.sql.gz.tmp`
  - проверить exit code pg_dump
  - если ненулевой — удалить `.tmp`, ошибка в лог, стоп
  - переименовать `.tmp` в финальное имя
  - если SFTP упал — стоп, не продолжать с другими
- очистить старые `.tmp` файлы

Ротация (новая операция):
- прочитать max_days и max_files из `.env`
- удалить файлы старше max_days
- из оставшихся оставить max_files равномерно по времени
- через SFTP listdir + remove
- ошибка ротации некритична

Логирование:
- полный traceback ошибок в лог-файл (для cron)
- статистика в терминал: сколько сервисов, успехов, ошибок

**Не делаем:** локальное хранение файлов, retry, restore, cron-настройку.

**Результат:** `db backup mytest` — файл на remote. Повторный запуск — ротация работает.

**Flow:** модуль `backup-remote`, definitions `[[backup-rotation]]`, `[[remote-backup-storage]]`

### Этап 9: Команда `db restore`

**Цель:** откатить базу к сохранённому состоянию — из локального дампа или с remote.

**Делаем:**

`db restore <name> [path]`:

Выбор источника (без path):
- показать локальные файлы из `dumps/` с датами и размерами
- получить список с remote через SFTP listdir
- объединить в нумерованный список
- оператор выбирает номер
- если выбран remote — скачать через SFTP во временную папку

Выбор источника (с path):
- использовать указанный файл напрямую

Восстановление:
- проверить что контейнер запущен
- если не запущен — сообщить, предложить запустить, завершить
- показать предупреждение: имя, текущий размер, файл бэкапа
- спросить подтверждение (`--force` пропускает)
- завершить активные подключения к базе
- DROP + CREATE database
- загрузить через psql (`.sql.gz` — распаковать на лету, `.sql` — напрямую)
- удалить временный файл если скачивался с remote

**Не делаем:** создание сервиса (ошибка: «сначала db add»), авто-бэкап перед restore.

**Результат:** `db dump mytest` → `db restore mytest` — выбрать из списка, данные восстановлены.

**Flow:** модуль `restore-service`

---

### Этап 10: Команда `db autobackup` и метрики

**Цель:** автоматизация — бэкапы по расписанию, метрики в SQLite.

**Делаем:**

`db autobackup install`:
- создать cron-задачу для `db backup` по расписанию из `.env`
- если метрики включены — создать cron для сбора метрик (каждые N минут)

`db autobackup uninstall`:
- удалить обе cron-задачи

`db autobackup status`:
- показать настройки из `.env`
- наличие cron-задач
- флаги бэкапов по каждому сервису

`db autobackup test`:
- запустить `db backup` прямо сейчас

`db autobackup logs`:
- показать хвост лог-файла

Новые операции:
- cron: обёртка над python-crontab (add/remove/find jobs)
- metrics: `docker stats --no-stream` → парсинг → SQLite
  - CPU %, память, сеть, диск
  - только running-контейнеры
  - ротации нет, хранится всё

**Не делаем:** команду просмотра метрик, веб-интерфейс, алертинг.

**Результат:** `db autobackup install` → `crontab -l` показывает задачи. `db autobackup test` — бэкап выполняется.

**Flow:** модуль `automation-control`, definition `[[metrics-db]]`

## Верификация

После каждого этапа — тесты через терминал.

Финальный цикл: `db add mytest` → правка .env + `db apply` → `db sizes` → `db dump mytest` → `db backup mytest` → `db restore mytest` → `db remove mytest --force`.
