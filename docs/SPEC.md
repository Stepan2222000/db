# Спецификация: система управления PostgreSQL-стеком

## 1. Обзор

CLI-инструмент `db` для управления PostgreSQL-сервисами внутри одного Docker Compose стека. Позволяет оператору создавать, удалять, бэкапить и восстанавливать отдельные PostgreSQL-инстансы через простые команды. Каждый сервис — отдельный контейнер с отдельным PostgreSQL.

Реализация на Python. Оператор — технический человек, знающий Docker и PostgreSQL.

---

## 2. Принципы

**Единственный источник правды** — файлы `.env.<service>`. Каждый сервис описан ровно в одном файле. Всё остальное вычисляется или генерируется из них.

**Одно имя везде.** Имя файла `.env.<service>` = имя сервиса = имя контейнера = имя базы данных. Никаких отдельных полей для container_name или db_name.

**`compose.yaml` — генерируемый артефакт.** Руками не редактируется. Перегенерируется при любом изменении. Первая строка: `# Автоматически сгенерировано. Не редактировать. Используй db apply.`

**Главный `.env` — только глобальные настройки.** Никакого реестра сервисов. Хранит настройки бэкапов, версию PostgreSQL, SSH-доступ к remote-серверу, метрики.

**Ошибки — fail fast.** Если что-то сломалось — полный traceback, ненулевой exit code, стоп. Без автооткатов. Без retry. Без fallback-логики. Оператор видит что сломалось и разбирается сам.

**Бэкапы стримятся напрямую на remote.** `pg_dump | gzip` → paramiko SFTP → remote-сервер. Локального файла не создаётся. Исключение — ручной `db dump`, который сохраняет файл в `dumps/`.

**Вывод: терминал + лог-файл.** Все команды пишут в терминал. Лог-файл ведётся только для `db backup` (потому что cron работает без терминала). При ошибках — полный traceback, без красивых обёрток.

## 3. Состояние системы

### Глобальный конфиг (`.env`)

Содержит настройки, общие для всего стека:

- Версия образа PostgreSQL (фиксированная, `18`)
- Дефолтный пароль для новых сервисов
- Настройки автобэкапов: расписание, включено/выключено, сжатие, таймаут
- SSH-доступ к remote-серверу: IP-адрес, логин, пароль
- Путь к хранилищу бэкапов на remote-сервере
- Параметры GFS-ротации (сколько хранить дней, недель, месяцев)
- Путь к лог-файлу и его максимальный размер
- Сбор метрик: включено/выключено (по умолчанию включён), интервал (по умолчанию 5 минут)

### Конфиг сервиса (`.env.<service>`)

По одному файлу на каждый сервис. Имя файла = имя сервиса = имя контейнера = имя базы данных. Одно имя везде.

Содержит:

- Пользователь PostgreSQL
- Пароль PostgreSQL
- Внешний порт

Опциональные поля:

- Максимальное количество подключений (по умолчанию 10000)
- Отключить автобэкап для этого сервиса (по умолчанию включён)
- Формат бэкапа: сжатый (`.sql.gz`) или без сжатия (`.sql`) — по умолчанию из глобального `.env`
- Лимит памяти контейнера (по умолчанию без ограничений — контейнер использует все свободные ресурсы)
- Лимит CPU контейнера (по умолчанию без ограничений)

Оператор может редактировать эти файлы руками, а затем запускать `db apply` для синхронизации. Также можно создать сервис вручную: создать `.env.<name>` и запустить `db apply` — compose перегенерируется, каталог `data/` создастся автоматически, контейнер поднимется.

### Генерируемый compose (`compose.yaml`)

Генерируется из всех `.env.*` файлов. Каждый сервис получает одинаковый блок:

- Образ `postgres:18`
- `container_name` = имя сервиса
- `env_file` указывает на `.env.<service>`
- Проброс порта из конфига на внутренний `5432`
- Том `./data/<service>:/var/lib/postgresql/data`
- `restart: unless-stopped`
- `shm_size` = RAM сервера * 0.9 / количество сервисов (пересчитывается при каждой генерации, читается из `/proc/meminfo`)
- `command: postgres -c max_connections=<N>` если задано в `.env.<service>`
- `deploy.resources.limits` (memory, cpus) если заданы в `.env.<service>`
- Healthcheck: `pg_isready`, interval 10s, timeout 5s, retries 3, start_period 30s

### Данные и файлы

- `data/<service>/` — PostgreSQL datadir. Монтируется как volume. Создаётся при добавлении сервиса (или при `db apply`, если каталога нет)
- `dumps/` — папка для ручных дампов (`db dump`)
- `metrics.db` — SQLite с метриками нагрузки контейнеров
- `.gitignore` — исключает `data/`, `dumps/`, `.env`, `.env.*`, `metrics.db`

## 4. Команды

### `db add <name> [--port X] [--password Y]`

Создать новый PostgreSQL-сервис.

**Шаги:**

1. Проверить имя сервиса — только буквы, цифры и подчёркивания
2. Если порт не указан — автоматически подобрать свободный, начиная с 5401, проверяя все существующие `.env.*`
3. Если порт указан — проверить диапазон (1024..65535) и что не занят другим сервисом
4. Проверить что `.env.<name>` ещё не существует
5. Создать `.env.<service>` с параметрами: пользователь `admin`, пароль из аргумента или дефолт из глобального `.env`, порт
6. Создать каталог `data/<service>/`
7. Перегенерировать `compose.yaml`
8. Показать итоговые параметры (имя, порт, пароль, версия postgres) и спросить подтверждение
9. `docker compose up <service> -d`
10. Дождаться готовности через `pg_isready` (до 60 секунд)
11. Вывести параметры подключения

**Ошибки:** если любой шаг не прошёл — сообщить что именно и остановиться. Не откатывать уже выполненные шаги.

---

### `db remove <name> [--force]`

Удалить сервис, контейнер и данные.

**Шаги:**

1. Проверить что `.env.<name>` существует. Если нет — показать список существующих сервисов и выйти
2. Показать что будет удалено: конфиг, контейнер (и его статус), каталог данных (и его размер)
3. Запросить подтверждения:
   - Согласие на удаление (y/n)
   - Повторный ввод точного имени сервиса
   - Ввод слова `DELETE` для подтверждения удаления данных
4. Остановить и удалить контейнер (если существует/запущен)
5. Удалить `.env.<service>`
6. Удалить `data/<service>/`
7. Перегенерировать `compose.yaml`

Флаг `--force` пропускает все подтверждения.

Бэкапы на remote-сервере не трогаем — они могут быть единственной копией данных.

---

### `db apply`

Синхронизировать состояние системы с конфигурацией. Основной сценарий: оператор руками отредактировал `.env.<service>` (сменил пароль, порт) и запускает `db apply`.

**Шаги:**

1. Создать `data/<service>/` для каждого `.env.*`, у которого нет каталога данных
2. Перегенерировать `compose.yaml` из всех `.env.*` (включая пересчёт `shm_size`)
3. Проверить конфликты портов между сервисами. Если конфликт — сообщить и остановиться
4. Показать что изменилось (diff между текущим и новым состоянием). Спросить подтверждение
5. `docker compose up -d` — Docker Compose сам определяет, какие контейнеры пересоздать (он отслеживает изменения в `env_file` и compose)
6. Для каждого запущенного контейнера — выполнить `ALTER ROLE` с паролем из `.env` через `docker exec`

**Почему ALTER ROLE нужен:** переменная `POSTGRES_PASSWORD` используется PostgreSQL только при первой инициализации (когда `data/` пуст). После этого пароль живёт внутри базы и env-переменная на него не влияет.

**Почему это безопасно:** `docker exec` запускает `psql` внутри контейнера через unix-сокет. PostgreSQL доверяет локальным подключениям (`trust` в `pg_hba.conf`), пароль для подключения не нужен.

**Идемпотентность:** `ALTER ROLE` с тем же паролем ничего не ломает. `docker compose up -d` без изменений не пересоздаёт контейнеры. Можно запускать `db apply` многократно.

**Примечание:** при добавлении или удалении сервиса `shm_size` пересчитывается для всех, поэтому Docker Compose пересоздаст все контейнеры.

---

### `db backup [name]`

Бэкап на remote-сервер. Используется cron-задачей и вручную.

**Шаги:**

1. Поставить lock (один процесс бэкапа одновременно)
2. Подключиться к remote-серверу по SSH через paramiko (IP, логин, пароль из `.env`). Если недоступен — стоп, ошибка в лог
3. Определить кандидатов: все сервисы из `.env.*` (или один конкретный по аргументу). Пропустить сервисы с отключённым бэкапом
4. Для каждого сервиса:
   - Найти запущенный контейнер
   - Стримить дамп напрямую на remote: `docker exec pg_dump | gzip` → paramiko SFTP → `<service>_YYYY-MM-DD_HH-MM.sql.gz.tmp` на remote
   - Проверить exit code от `pg_dump`. Если ненулевой — удалить `.tmp`, записать ошибку в лог, стоп
   - Переименовать `.tmp` в финальное имя
   - Если SFTP упал — стоп, ошибка в лог, не продолжать со следующими сервисами
5. GFS-ротация на remote через то же SFTP-соединение (listdir + remove)
6. Вывести статистику (сколько сервисов, успехов, ошибок)
7. Всё логируется в файл и терминал. Ошибки — полный traceback

**Формат имени файла:** `<service>_<YYYY-MM-DD_HH-MM>.sql.gz` (или `.sql`)

**Структура на remote:** `<remote_path>/<hostname>/<service>/` — hostname определяется автоматически.

---

### `db dump <name>`

Ручной дамп, файл остаётся локально.

**Шаги:**

1. Найти запущенный контейнер для сервиса
2. Проверить свободное место: запросить `pg_database_size()`, умножить на 0.3 для gzip или 1.0 для plain. Если не хватает — сообщить сколько нужно и сколько есть, стоп
3. Стримить дамп: `docker exec pg_dump | gzip > dumps/<service>_YYYY-MM-DD_HH-MM.sql.gz`
4. Сообщить путь к файлу и его размер

---

### `db restore <name> [path]`

Восстановить базу из бэкапа. Разрушительная операция — текущие данные удаляются.

**Без аргумента `path` — интерактивный выбор:**

1. Показать локальные файлы из `dumps/` (если есть), с датами и размерами
2. Подключиться к remote по SSH (paramiko), получить список файлов с датами и размерами
3. Показать объединённый нумерованный список
4. Оператор выбирает номер
5. Если выбран remote-файл — скачать через paramiko SFTP во временную папку

**С аргументом `path`:** использовать указанный локальный файл.

**Восстановление:**

1. Проверить что контейнер сервиса запущен. Если нет — сообщить, предложить запустить, завершить
2. Показать предупреждение: имя сервиса, текущий размер БД, файл бэкапа. Спросить подтверждение
3. Завершить активные подключения к базе
4. Удалить базу и создать заново (DROP + CREATE)
5. Загрузить дамп через `psql` (`.sql.gz` распаковать через gzip, `.sql` напрямую)
6. Удалить временный файл (если скачивался с remote)

Поддерживает `.sql` и `.sql.gz` — определяется по расширению файла.

Флаг `--force` пропускает подтверждение.

Дампы совместимы вперёд между версиями PostgreSQL (дамп от 17 загрузится в 18).

---

### `db sizes`

Показать размер каждой базы данных.

**Вывод:** имя сервиса и размер в ГБ. Размер берётся из PostgreSQL (`pg_database_size`) для запущенных сервисов, с диска (`du`) для остановленных.

При пустом стеке (нет сервисов) — сообщить «нет сервисов».

---

### `db autobackup install|uninstall|status|test|logs`

Управление автоматическими бэкапами и сбором метрик через cron.

- `install` — создать cron-задачу бэкапов с расписанием из глобального `.env`. Если в `.env` включён сбор метрик — также создать cron метрик (каждые N минут)
- `uninstall` — удалить cron-задачи (бэкапы + метрики)
- `status` — показать настройки из `.env`, наличие cron-задач, настройки бэкапов по каждому сервису
- `test` — запустить `db backup` вручную прямо сейчас
- `logs` — показать хвост лог-файла бэкапов

## 5. Бэкап-политика

### Два режима

| | Автоматический (`db backup`) | Ручной (`db dump`) |
|---|---|---|
| Куда | Remote-сервер через paramiko SFTP | Локально в `dumps/` |
| Как | Стрим напрямую, без локального файла | Запись в локальный файл |
| Когда | cron или вручную | Только вручную |
| Ротация | GFS на remote | Нет (оператор сам) |

### GFS-ротация

Выполняется нашим скриптом через paramiko SFTP на remote-сервере после каждой успешной отправки.

Правила (дефолтные значения настраиваются в глобальном `.env`):

- Хранить все бэкапы за последние **7 дней**
- Хранить воскресные бэкапы за последние **4 недели**
- Хранить бэкапы первого числа месяца за последние **3 месяца**
- Всё остальное — удалить

Итого: максимум ~14 файлов на сервис.

### Структура на remote

```
<remote_path>/
  <hostname>/
    <service_a>/
      service_a_2026-04-12_02-00.sql.gz
      service_a_2026-04-11_02-00.sql.gz
      ...
    <service_b>/
      service_b_2026-04-12_02-00.sql.gz
      ...
```

`<hostname>` определяется автоматически — позволяет нескольким серверам слать бэкапы на один remote без конфликтов.

### Безопасность передачи

Дамп стримится во временный файл `.tmp` на remote. После завершения — переименовывается в финальное имя. Если стрим оборвался — `.tmp` остаётся, GFS его не видит (работает только с `.sql.gz` / `.sql`). При следующем запуске бэкапа старые `.tmp` удаляются.

### Обработка ошибок

- SSH/SFTP недоступен — стоп на этапе подключения, дамп не начинается
- SFTP упал посреди передачи — стоп, `.tmp` остаётся, ошибка в лог
- `pg_dump` вернул ненулевой exit code — `.tmp` удаляется, ошибка в лог, стоп
- Ротация по SFTP упала — не критично, файлы накопятся, почистятся при следующем прогоне

## 6. Метрики

Фоновый сбор нагрузки контейнеров для последующего анализа.

- **Что собирается:** CPU %, использование памяти, сетевой и дисковый I/O по каждому контейнеру (через `docker stats --no-stream`)
- **Куда:** SQLite база `metrics.db` в корне проекта
- **Как часто:** каждые 5 минут (настраивается в `.env`)
- **Управление:** cron ставится/снимается через `db autobackup install/uninstall`
- **Включение:** флаг в `.env` (по умолчанию включён)
- **Просмотр:** оператор подключается к SQLite сам — отдельной команды для просмотра нет
- **Ротация:** нет, данные хранятся бессрочно
- **Собирается только для запущенных контейнеров**

## 7. Технические решения

**Язык:** Python 3.10+

**Зависимости** (устанавливаются через `requirements.txt`):

- `typer` — CLI-фреймворк с подкомандами и автогенерацией `--help`. Включает `rich` для вывода. Документация: https://typer.tiangolo.com/
- `pyyaml` — генерация compose.yaml через `yaml.dump()`. Документация: https://pyyaml.org/wiki/PyYAMLDocumentation
- `paramiko` — SSH/SFTP-клиент. Подключение по паролю. Используется для: передача бэкапов, GFS-ротация, список файлов для restore, скачивание при restore. Одно подключение на весь процесс. Документация: https://docs.paramiko.org/en/stable/

**Стандартная библиотека:**

- `sqlite3` — хранение метрик
- `subprocess` — вызов `docker compose`, `docker exec`, `crontab`
- Парсинг `.env` файлов — руками (простые KEY=VALUE)
- Цветной вывод — ANSI-коды

**Внешние инструменты** (вызываются через subprocess):

- `docker compose` — управление контейнерами
- `pg_dump` / `psql` — через `docker exec` внутри контейнера
- `crontab` — управление расписанием

**Генерация compose.yaml:** из Python dict через `pyyaml`. `shm_size` вычисляется из `/proc/meminfo` (total RAM * 0.9 / количество сервисов).

**ALTER ROLE:** `docker exec <container> psql -U <user> -c "ALTER ROLE <user> PASSWORD '<password>'"`. Через unix-сокет, trust-аутентификация.

**Docker Compose и env_file:** `docker compose up -d` автоматически обнаруживает изменения в `env_file` и пересоздаёт контейнеры. Если ничего не изменилось — не трогает. Проверено экспериментально на Docker Compose v5.

**CLI запускается из каталога проекта.** Если `.env` или `compose.yaml` не найден — ошибка.

## 8. Что не делаем

- **Переименование сервиса** — вместо этого: `db remove` + `db add` + `db restore`
- **Переименование пользователя** — та же логика
- **Обновление версии PostgreSQL** — через `db dump` + `db remove` + поменять версию в `.env` + `db add` + `db restore`
- **`db stop` / `db start`** — оператор использует `docker compose stop/start` напрямую
- **Orphan-контейнеры** — отмечено как потенциальная проблема, не проектируем сейчас
- **Команда setup/init** — оператор сам заполняет `.env`
- **Email/webhook-уведомления** — лог-файл достаточен
- **Ресурсные лимиты по умолчанию** — контейнеры используют все свободные ресурсы. Лимиты задаются явно в `.env.<service>`
- **Команда для просмотра метрик** — оператор открывает SQLite сам
- **Автооткат при ошибках** — fail fast, оператор разбирается
- **Retry** — если упало, значит что-то не так
- **Защита от параллельных операций** — кроме lock для backup. Один оператор на сервере
- **Локальная ротация бэкапов** — их просто нет локально (кроме ручных dump)
