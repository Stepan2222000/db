Ниже — **зафиксированное словами описание** системы трекинга активностей, реализованной как **OpenClaw-плагин**. Это не код и не конфиг — именно **смысл, правила работы и архитектурные решения**.

---

## 1) Цель системы и общий принцип

Персональная система трекинга "что я делал" (активности + события). Система для **одного пользователя**. Основной интерфейс — **один Telegram direct chat с ботом**.

Система должна:

* регулярно (примерно раз в 30 минут) проверять, что происходило, и **пинговать**, если непонятно чем пользователь занимается; промежутки времени между каждым пингом у нас указываются в конфиге то есть к 30 минут к примеру или 50
* записывать только **изменения + важное**, а не каждое сообщение;
* переживать перезапуски и обрывы сессий, оставаясь консистентной;
* работать полностью внутри одного чата с ботом, не плодя отдельные контексты трекинга.

Важный вывод: **"история чата" и "долговечная память" — разные сущности**. Долговечные заметки нужны, чтобы состояние трекера не зависело от того, что модель "вспомнит" модель а строго фиксировалось

---

## 2) Что мы НЕ делаем

Сознательные упрощения:

* **Не храним raw-слой** (оригиналы голосовых/фото). OpenClaw сам обрабатывает голос и изображения.
* **Не храним фото**. Если фото нужно "для смысла" — в память попадает только текстовый факт.
* **Не вводим строгую систему entities** на старте (отдельные карточки людей/проектов). Entities добавим позже, если заболит.
* **Не делаем пользовательских команд** (/mode, /status, /export). Всё через естественный язык. Агент сам решает что считать изменением.
* **Не закладываем обработку ошибок** на уровне дизайна. Edge cases решаются по факту.
* **Бэкапы** — не нужны сейчас.

## 3) Почему плагин, а не skill

### 3.1. Проблема: разрозненная реализация

Раньше трекер был собран из отдельных кусков:
* **Skill** (`SKILL.md`) — текстовые инструкции для LLM
* **Cron-задача** — отдельная запись в `cron/jobs.json`
* **Ручные systemPrompt** — записи в `openclaw.json` для нужной сессии/чата
* **Runtime-файлы** — `status.md`, дневные файлы в `memory/`

Чтобы "установить" трекер — нужно было руками создать skill, добавить cron-задачу и прописать systemPrompt'ы в конфиге. Чтобы "удалить" — руками обойти 4+ мест. Ничего не связано между собой.

### 3.2. Решение: OpenClaw Plugin

Плагин — npm-пакет (TypeScript, загружается через jiti без компиляции), который **бандлит всё в одну единицу**:

* **AGENTS.md** (статичные инструкции) + **хук `before_prompt_build`** (периодический статус) заменяют ручные systemPrompt'ы.
* **Tools** заменяют прямую работу LLM с файлами. Вместо "прочитай YAML, распарси, обнови поле" — структурированные вызовы.
* **CLI setup/remove** управляет cron-задачей (inject в основной чат), записью в AGENTS.md и конфигом плагина.
* **Конфигурация** — единый блок в `openclaw.json` под `plugins.entries`.

Установка = `openclaw tracker setup`. Удаление = `openclaw tracker remove`. Всё в одном месте.

### 3.3. Что НЕ меняется

Плагин не переносит **семантическую логику** из LLM в код. LLM по-прежнему решает:
* "Это смена активности или просто разговор?"
* "Как сформулировать пинг?"
* "Что считать значимым изменением?"
* "Пользователь передумал спать?"

Плагин берёт на себя **механику**: парсинг файлов, форматирование записей и периодическое подмешивание статуса в контекст. Арифметику (тихие часы, freshness, таймер игнора) агент считает сам по правилам из AGENTS.md.

## 4) Архитектура: Telegram Direct Chat + Cron

### 4.1. Как устроен интерфейс

Система живёт в **одном Telegram direct chat с ботом**. Один агент, одна сессия, одна общая память (файлы), один основной диалог для всего — и для трекинга, и для cron-пингов.

Архитектура OpenClaw:
* Один агент `main`, один workspace (`agents.defaults.workspace`), один `MEMORY.md`
* Основной direct chat — единственный рабочий диалог, сессия `agent:main:telegram:direct:<chatId>`
* Cron запускается **в той же сессии** (через `sessionTarget: "session:agent:main:telegram:direct:<chatId>"`) — агент видит полную историю разговора
* Долгосрочная память (файлы) — `memory/status.md` и `memory/YYYY-MM-DD.md`
* `memory_search` работает из основного чата

### 4.2. Один контекст трекинга: как работает direct chat

Весь трекинг живёт в **одном основном чате**. Именно сюда приходят пинги каждые 30 минут, здесь агент активно спрашивает чем занимаешься, следит за молчанием и ведёт хронологию дня.

Ключевая идея: агент не ищет фразы "начал" или "закончил". Он смотрит на смысл разговора в этом одном чате. Если пользователь обсуждает баги парсера — значит работает над парсером. Если пишет, что закончил одну задачу и переключился на другую, это тоже обновление трекера.

**Добавление активностей** — без лишней бюрократии, когда из контекста достаточно понятно, что активность началась или сменилась. Если агент ошибся — пользователь поправит.

**Закрытие активностей** не должно происходить механически по любому сдвигу темы. Если контекст неоднозначный — лучше уточнить.

**Не является активностью**: приветствия, шутки, мемы, абстрактное любопытство. В спорных случаях — спросить.

**Принцип: один чат — один контекст трекинга.**

### 4.3. Как агент знает что он трекер

Два механизма, дополняющих друг друга:

* **AGENTS.md** — статичные инструкции трекера (роль, правила, tools, тихие часы, правило 3 часов). Записываются плагином при `openclaw tracker setup`. Агент видит их **в каждом сообщении** — это часть system prompt.
* **Хук `before_prompt_build`** — периодически (раз в 5 сообщений) инжектит текущий статус из `status.md`: mode, активности, ping_pending, updated_at. Напоминает агенту актуальное состояние трекера.

Cron работает **в той же сессии**, поэтому отдельного контекста для cron не нужно. Cron-сообщение — обычный agent turn, агент видит историю и AGENTS.md, и сам решает нужен ли пинг.

## 5) Компоненты плагина

### 5.1. Tools (AI-инструменты)

Плагин регистрирует tools через [`api.registerTool()`](https://docs.openclaw.ai/plugins/building-plugins#registering-agent-tools). Имена в snake_case (конвенция OpenClaw). LLM не парсит YAML руками — вызывает tool и получает/отправляет структурированные данные.

| Tool | Что делает | Кто вызывает |
|------|-----------|-------------|
| `tracker_status` | Читает `status.md`, возвращает структурированный объект (mode, active, ping_pending, updated_at и т.д.) | Агент |
| `tracker_update` | Обновляет `status.md` (новая активность, смена режима, сброс пинга). Принимает partial update — read-modify-write атомарно. Автоматически проставляет `updated_at`. | Агент |
| `tracker_log` | Записывает строку в дневной файл `memory/YYYY-MM-DD.md`. Автоформатирование: время (таймзона из конфига). Создаёт файл и секцию `## Трекинг` если не существует. | Агент |

**Как tools получают доступ к файлам:** Плагин использует Node.js `fs/promises` напрямую (плагины запускаются in-process с Gateway). Путь к workspace берётся через `api.runtime.agent.resolveAgentWorkspaceDir()`. Tools читают/пишут `<workspace>/memory/status.md` и `<workspace>/memory/YYYY-MM-DD.md`.

Пример: cron-сообщение приходит в основной чат, агент решает пинговать:
```
1. tracker_status() → { mode: "normal", active: [...], updated_at: "45 мин назад", ping_pending: false }
2. Агент видит что статус устарел, формулирует текст пинга
3. tracker_update({ ping_pending: true, last_ping_at: "now" })
4. tracker_log({ text: "Пинг отправлен" })
```

Пример обновления из обычного разговора в основном чате:
```
1. Пользователь: "закончил авторизацию, перехожу к тестам"
2. Агент: tracker_update({ active: [{ name: "Тестирование", started_at: "now" }] })
3. Агент: tracker_log({ text: "Закончил авторизацию, начал тестирование" })
4. Агент: "Записал: закончил авторизацию, начал тесты"
```

### 5.2. Хук `before_prompt_build` (периодический)

Хук вызывается перед каждым ответом агента, но **инжектит контекст не каждый раз**, а периодически — раз в N сообщений (N из конфига: `remindEveryMessages`, дефолт 5). Это экономит токены: агент помнит контекст через историю (lossless-claw сохраняет разговор полностью), а хук периодически обновляет актуальный статус.

Логика:
1. Получает `messages` (историю сессии).
2. Если `messages.length % remindEveryMessages === 0` или первое сообщение после рестарта gateway → читает `status.md` и инжектит.
3. Иначе — ничего не возвращает, агент работает "по памяти".

Инжектируемый контекст — тег `<activity-tracker-status>`:
* Текущее состояние из status.md (mode, активности с started_at, ping_pending, updated_at)
* Короткое напоминание: "Работай по инструкциям трекера из AGENTS.md"

Возвращает `{ appendSystemContext: "..." }` (не prependContext — чтобы не мешать другим инъекциям).

### 5.2.1. AGENTS.md — статичные инструкции трекера

CLI `openclaw tracker setup` дописывает в конец `AGENTS.md` блок между маркерами:

```markdown
<!-- activity-tracker-start -->
## Activity Tracker

Ты — трекер активностей. Правила работы:

**Tools:** tracker_status, tracker_update, tracker_log. Используй их для чтения/записи состояния.

**Трекинг:** При каждом сообщении оценивай, изменилась ли активность. Если да — tracker_update + tracker_log. Не будь навязчив, не спрашивай если очевидно.

**Пинги (cron):** Периодически приходит cron-сообщение "Проверь статус трекера". Вызови tracker_status, посмотри updated_at. Если прошло больше {pingIntervalMinutes} минут и не тихие часы и не sleeping — спроси чем занимается. Если статус свежий — ничего не делай.

**Сон:** "Ложусь спать" → tracker_update(mode: sleeping, active: [], sleep_started_at: "now") + tracker_log. Любое сообщение после → проснулся.

**Тихие часы:** {quietHours}. Не пингуй в это время.

**Правило игнора:** Если ping_pending и прошло > {ignoreTimeoutHours}ч (вычитая тихие часы) — зафиксируй игнор через tracker_log, сбрось ping_pending.
<!-- activity-tracker-end -->
```

Значения `{pingIntervalMinutes}`, `{quietHours}`, `{ignoreTimeoutHours}` подставляются из конфига при setup.

CLI `openclaw tracker remove` удаляет блок между маркерами. Путь к файлу: `api.runtime.agent.resolveAgentWorkspaceDir()` + `/AGENTS.md`.

### 5.3. CLI-команды

Плагин регистрирует CLI через `api.registerCli()`:

| Команда | Что делает |
|---------|-----------|
| `openclaw tracker setup` | Записывает конфиг плагина в openclaw.json, записывает инструкции трекера в AGENTS.md, создаёт cron-задачу через `openclaw cron add`, создаёт начальный status.md |
| `openclaw tracker remove` | Удаляет cron-задачу через `openclaw cron rm`, удаляет блок трекера из AGENTS.md, отключает плагин. Данные (status.md, дневные файлы) **не удаляет** |
| `openclaw tracker status` | Показывает текущий конфиг, содержимое status.md, статус cron-задачи |

### 5.4. Конфигурация

Единый блок в `openclaw.json` под `plugins.entries`:

```json
{
  "plugins": {
    "entries": {
      "openclaw-activity-tracker": {
        "enabled": true,
        "config": {
          "chatId": "123456789",
          "timezone": "Europe/Samara",
          "quietHours": "23:00-08:00",
          "pingIntervalMinutes": 30,
          "ignoreTimeoutHours": 3,
          "remindEveryMessages": 5
        }
      }
    }
  }
}
```

Поля конфига:

| Поле | Обязательно | Дефолт | Назначение |
|------|------------|--------|-----------|
| `chatId` | да | — | ID direct chat в Telegram |
| `timezone` | нет | `"Europe/Samara"` | Таймзона для форматирования записей |
| `quietHours` | нет | `"23:00-08:00"` | Часы, когда cron не пингует |
| `pingIntervalMinutes` | нет | `30` | Интервал между пингами (мин) |
| `ignoreTimeoutHours` | нет | `3` | Через сколько часов молчания фиксировать игнор |
| `remindEveryMessages` | нет | `5` | Раз в сколько сообщений хук инжектит статус трекера |
| `debug` | нет | `false` | Расширенное логирование |

`status.md` остаётся для **runtime-состояния** (текущие активности, mode, ping_pending). Настройки — только в конфиге.

### 5.5. Cron-задача

Документация OpenClaw говорит: ручные правки `cron/jobs.json` безопасны только при остановленном Gateway. Правильный способ — [`openclaw cron add/edit/remove`](https://docs.openclaw.ai/cli#cron).

CLI setup создаёт задачу через вызов:
```bash
openclaw cron add \
  --name "tracker-ping" \
  --every "30m" \
  --session "session:agent:main:telegram:direct:<chatId>" \
  --message "Проверь статус трекера. Если давно не было обновлений — спроси чем занимаюсь."
```

Ключевой момент: `sessionTarget: "session:agent:main:telegram:direct:<chatId>"` — cron запускает agent turn **в существующей сессии** основного чата. Агент видит полную историю разговора, инструкции из AGENTS.md и сам решает нужен ли пинг. Отдельная модель не нужна — используется модель основного агента. Delivery-параметры (`--announce`, `--channel`, `--to`) не нужны — агент уже в нужном чате.

CLI remove удаляет задачу через `openclaw cron rm <jobId>`.

Плагин при `register()` **не трогает** cron — только CLI.

### 5.6. Манифест плагина (предварительный)

Файл `openclaw.plugin.json` обязателен для любого плагина:

```json
{
  "id": "openclaw-activity-tracker",
  "name": "Activity Tracker",
  "description": "Personal activity tracking with periodic pings",
  "kind": "tracker",
  "configSchema": {
    "type": "object",
    "additionalProperties": false,
    "properties": {
      "chatId": { "type": "string" },
      "timezone": { "type": "string" },
      "quietHours": { "type": "string" },
      "pingIntervalMinutes": { "type": "number" },
      "ignoreTimeoutHours": { "type": "number" },
      "remindEveryMessages": { "type": "number" },
      "debug": { "type": "boolean" }
    },
    "required": ["chatId"]
  }
}
```

`kind: "tracker"` — свой тип. Слот не нужен (трекер не конкурирует с другими плагинами за эксклюзивную категорию, в отличие от memory-плагинов).

## 6) Память: структура файлов

Два главных уровня: "срез сейчас" и "история за день".

### 6.1. `memory/status.md` — текущее состояние

`status.md` — это **snapshot** трекера: что считается активным прямо сейчас, какой режим, когда последний апдейт, есть ли ожидание ответа на пинг.

Формат: **YAML frontmatter** в .md файле. Читается и пишется через tools (`tracker_status` / `tracker_update`), а не руками LLM. Парсинг YAML — через библиотеку `js-yaml` внутри плагина.

Поля:

* `updated_at` — когда статус **содержательно** менялся последний раз (активность открылась/закрылась). ISO 8601 с таймзоной. Обязательно.
* `active` — список текущих активностей (0..N), каждая с `name` и `started_at`. Параллельные активности поддерживаются.
* `mode` — нормальный режим или спящий (`normal` / `sleeping`)
* `last_ping_at` — когда последний раз был отправлен пинг
* `ping_pending` — ждём ли ответа на пинг (true/false)
* `sleep_started_at` — когда лёг спать (только при `mode: sleeping`)

Пример:

```yaml
---
updated_at: "2026-02-14T18:30:00+04:00"
mode: normal
active:
  - name: "Кодинг парсера"
    started_at: "2026-02-14T16:00:00+04:00"
last_ping_at: "2026-02-14T18:00:00+04:00"
ping_pending: false
sleep_started_at: null
---
```

Важно: поле `last_user_message_at` **не нужно**. Трекер ориентируется на `updated_at` — момент содержательного изменения статуса, а не факт любого сообщения.

`quiet_hours` **убран** из status.md — это статичная настройка в конфиге плагина, а не runtime-состояние.

**Холодный старт**: если status.md не существует, `tracker_status` возвращает дефолты (`mode: normal`, `active: []`, `ping_pending: false`). `tracker_update` создаёт файл при первой записи.

### 6.2. `memory/YYYY-MM-DD.md` — дневной файл

Дневные файлы хранятся **прямо в `memory/`**. Трекер дописывает в них свои записи через `tracker_log`. Агент также может вручную писать в эти файлы через встроенные инструменты OpenClaw.

Записывается через tool `tracker_log`:
* история смен состояния/активностей — с **таймстемпами**;
* отметки о пингах (отправлен, отвечен, проигнорирован);
* важные наблюдения, но только действительно значимое.

Формат: отдельная секция `## Трекинг` с таймстемпами по таймзоне из конфига. Каждая строка — `- HH:MM — Действие`. Tool `tracker_log` автоматически добавляет время.

Пример:

```markdown
## Трекинг

- 08:15 — Проснулся (сон: вчера 23:30 — 08:15, ~8.75ч)
- 09:00 — Начал: кодинг парсера
- 12:00 — Пинг отправлен
- 12:05 — Ответ на пинг: "всё ещё кодю"
- 14:00 — Закончил кодинг (5ч), начал: обед
```

`status.md` хранит "сейчас". Дневной файл хранит "как менялось". Записи в дневном файле **остаются навсегда**.

Длительность активности: агент **сам считает** продолжительность — tool `tracker_status` возвращает `started_at` каждой активности, агент видит текущее время.

Стык суток: запись идёт в файл того дня, когда событие закрытия произошло.

Ретроспективные записи: `tracker_log` принимает опциональный параметр `date` (формат `YYYY-MM-DD`), чтобы записать событие в файл прошлого дня. При ретроспективной записи tool автоматически добавляет пометку `[retrospective]` перед текстом. Пример: `- 14:00 — [retrospective] Обедал`. Это отличает записи, сделанные задним числом, от записей в реальном времени.

### 6.3. `MEMORY.md`

Короткий "профиль" и правила. OpenClaw индексирует этот файл — он всегда в контексте агента.

## 7) Cron: пинги и логика молчания

### 7.1. Почему cron session-based

OpenClaw поддерживает несколько вариантов cron: `isolated` (свежая сессия), `main` (heartbeat), `session:<key>` (в существующей сессии). Трекер использует **session-based cron** — `sessionTarget: "session:agent:main:telegram:direct:<chatId>"`.

Почему:
* Агент видит **полную историю разговора** — не нужен отдельный tool для проверки "нужен ли пинг"
* Агент видит **AGENTS.md** с инструкциями трекера — знает правила
* Нет отдельного cron-контекста, одна сессия, простая архитектура
* Агент сам решает на основе контекста — умнее, чем механическая проверка

### 7.2. Как работает cron-пинг

Recurring cron job запускается каждые N минут (из конфига `pingIntervalMinutes`). Механика:

1. OpenClaw запускает **agent turn в существующей сессии** основного чата
2. Агент получает cron-сообщение: "Проверь статус трекера. Если давно не было обновлений — спроси чем занимаюсь."
3. Агент видит историю разговора + инструкции из AGENTS.md + (если попал на N-е сообщение) свежий статус из хука
4. Агент сам решает: вызывает `tracker_status()`, смотрит `updated_at`, `mode`, `ping_pending`, тихие часы
5. Если пинг нужен → пишет пользователю, вызывает `tracker_update(ping_pending: true, last_ping_at: "now")` и `tracker_log("Пинг отправлен")`
6. Если не нужен (статус свежий, sleeping, тихие часы, ping_pending) → ничего не делает

Пинг — **текстовое сообщение** в основной чат. Без кнопок. Просто текст вроде "Чем занимался последние полчаса?".

### 7.3. Логика принятия решения (агент)

Правила прописаны в AGENTS.md. Агент проверяет через `tracker_status()`:

1. Если `mode: sleeping` → не пинговать (пользователь спит)
2. Если сейчас тихие часы (из AGENTS.md) → не пинговать
3. Если `updated_at` свежий (менее `pingIntervalMinutes` минут назад) → не пинговать (статус актуален)
4. Если `ping_pending: true` → проверить таймер игнора (см. §9)
5. Иначе → пинговать

Агент также видит историю разговора — если 5 минут назад общался с пользователем, он и без `updated_at` понимает что пинг не нужен.

### 7.4. Что происходит после пинга

* Пользователь **ответил** → агент обрабатывает ответ, вызывает `tracker_update` и `tracker_log`, сбрасывает `ping_pending`
* Пользователь **молчит** → следующий cron видит `ping_pending: true` и не шлёт новый пинг (не спамит)
* Пользователь **сам обновил статус** → `updated_at` обновился → `ping_pending` сбрасывается

## 8) Сон

Когда пользователь говорит "ложусь спать" (в основном чате):

1. Агент закрывает все текущие активности через `tracker_update`
2. Устанавливает `mode: sleeping`, `sleep_started_at: <время>`, `active: []`
3. Записывает через `tracker_log`: "Лёг спать"
4. Cron видит `mode: sleeping` → не пингует

Когда пользователь пишет что-либо после сна:

1. Агент убирает `mode: sleeping` через `tracker_update`
2. Записывает через `tracker_log` в дневной файл **нового дня**: "Проснулся (сон: 23:30 — 08:00, ~8.5ч)"
3. Система возвращается к обычному режиму

Если пользователь передумал ("не засну", "ещё посижу") — агент убирает `mode: sleeping`.

Таймстемп сна **остаётся навсегда** в дневном файле — это часть истории.

## 9) Правило "3 часа — считаем игнором"

Отсчёт **от момента отправки пинга** (`last_ping_at`). Порог — из конфига (`ignoreTimeoutHours`).

Если прошло N часов и:
* ответа на пинг не было, И
* статус не обновлялся в основном чате

Тогда:
* фиксируем через `tracker_log`: "Пинг от [время] проигнорирован (Nч без ответа). Интервал — нет данных."
* сбрасываем `ping_pending: false` через `tracker_update`
* **не придумываем** активность. Честная пустота лучше ложных фактов.

### Взаимодействие с тихими часами

Тихие часы **замораживают** таймер. Агент вычитает из прошедшего времени часы, попавшие в тихий период (правила в AGENTS.md).

Пример:
* Пинг в 22:30, ответа нет
* 23:00–08:00 — тихие часы (9 часов не считаются)
* 08:30 — cron: прошло 10ч реального времени, но эффективно только 1ч. Таймер продолжается.
* 10:30 — эффективно прошло 3ч → фиксируем игнор

### Взаимодействие со сном

Если пользователь сказал "ложусь спать" → `mode: sleeping`. Если до этого был незакрытый пинг — `ping_pending` сбрасывается (мы знаем что человек пошёл спать, это не игнор).

## 11) Lossless-claw вместо compaction

Стандартный compaction OpenClaw отключён. Используется плагин **lossless-claw** (`@martian-engineering/lossless-claw`) — DAG-based conversation summarization, которая сохраняет историю разговора полностью.

Это означает:
* Агент **надёжно помнит** контекст через историю сессии — не нужно инжектить статус на каждое сообщение
* Дневные файлы `memory/YYYY-MM-DD.md` остаются как **долговечное хранилище** для структурированной хронологии — это другой формат данных, не дублирование истории чата
* Агент также вручную пишет в дневные файлы через встроенные инструменты OpenClaw — lossless-claw не отменяет эту привычку, дневные файлы остаются
* Правило: записывать важное **по факту изменений** (при каждой смене активности, при каждом пинге)

## 14) Решённые вопросы

| Вопрос | Решение |
|--------|---------|
| Skill vs Plugin | **Plugin**. Бандлит tools + hooks + cron + конфиг. Установка/удаление одной командой. |
| systemPrompt'ы | Статичные инструкции в AGENTS.md (setup записывает, remove удаляет) + периодический хук `before_prompt_build` для обновления статуса. |
| Работа с файлами | Через tools (`tracker_status`, `tracker_update`, `tracker_log`), а не прямой парсинг LLM. |
| Файловый доступ из плагина | `fs/promises` напрямую. Workspace path через `api.runtime.agent.resolveAgentWorkspaceDir()`. |
| `updated_from` | Поле убрано: для single-chat версии оно не нужно. |
| quiet_hours | В конфиге плагина (статичная настройка), а не в status.md (runtime). |
| Холодный старт | CLI setup создаёт начальный status.md. Если status.md пропал — tool `tracker_status` возвращает дефолты, `tracker_update` создаёт файл при записи. |
| Формат status.md | YAML frontmatter. Парсится через `js-yaml` внутри tools. |
| Формат дневного файла | Секция `## Трекинг` с записями `- HH:MM — Действие`. Пишется через `tracker_log`. |
| Heartbeat vs Cron | **Cron session-based**. `sessionTarget: "session:agent:main:telegram:direct:<chatId>"` — inject в существующую сессию, агент видит полный контекст. |
| `tracker_should_ping` | **Убран**. Агент решает сам по правилам из AGENTS.md, видит историю и `tracker_status()`. |
| AGENTS.md | Статичные инструкции трекера записываются setup между маркерами, удаляются remove. |
| Хук | Периодический (раз в `remindEveryMessages` сообщений), инжектит текущий статус из status.md. |
| workspacePath | Убран из конфига. Используется `api.runtime.agent.resolveAgentWorkspaceDir()`. |
| Compaction | Lossless-claw (`@martian-engineering/lossless-claw`), стандартный компакт отключён. |
| Plugin API | `api.config` (полный конфиг gateway, не `api.pluginConfig`). Хук: `before_prompt_build` (не `before_agent_start` — legacy). |
| Регистрация cron | CLI setup через `openclaw cron add` (не ручная запись в jobs.json — документация: "manual edits only safe when Gateway stopped"). |
| kind плагина | `"tracker"` — свой тип. Слот не нужен. |
| Персона агента | Из SOUL.md: русский язык, по-дружески, коротко. Пинги — лаконичные. |
| Ретроспективные записи | `tracker_log` с пометкой `[retrospective]` в нужный день. |
| userTimezone | В конфиге плагина (`timezone`). |
| Конкурентный доступ | `tracker_update` делает read-modify-write. Для однопользовательской системы с редкими параллельными записями — достаточно. Lock-файл — на будущее если заболит. |
| TypeScript runtime | Плагины загружаются через jiti, компиляция не нужна. ESM (ES2022). |
| Telegram delivery | Формат: direct `chatId`. |
