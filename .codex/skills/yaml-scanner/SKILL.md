name: yaml-scanner
description: >-
  Сканирует существующий код и генерирует YAML-документ логики системы
  в формате purpose → definitions → modules → steps → links.
  Используй, когда нужно извлечь из кодовой базы структурированное описание:
  что система делает, из каких частей состоит, как устроены её runtime flows.
  Пользователь указывает директорию или конкретные файлы для анализа.
  Не используй для генерации кода из YAML (это yaml-author),
  для ревью кода, рефакторинга или написания тестов.

## Входы

- Путь к директории или список файлов для сканирования (обязательно).
- Опциональный фокус: "просканируй flow оплаты", "опиши API-слой" — сужает анализ.
- Если есть существующий YAML — режим дополнения: сканирует код и обновляет YAML.

## Цель

Из кода извлечь YAML-документ в том же формате, что создаёт yaml-author.
Результат должен быть полезен как спецификация системы — читаемый граф flow,
а не механический слепок файловой структуры.

Два ключевых принципа:

1. **Абстрагируй механику, сохраняй поведение.** Убирай имена функций, паттерны блокировок, внутренние переменные. Но сохраняй все состояния, переходы между ними и правила, по которым система принимает решения. Если в коде есть N разных сценариев реакции на событие — все N должны быть видны в YAML (в шагах или definitions), даже если ты не упоминаешь имя функции.

2. **Описывай на языке предметной области.** Ты видишь функции, классы, переменные — но в YAML пишешь на языке системы. Примеры перевода с кода на язык системы — в `references/abstraction-guide.md`. Прочитай его перед первой генерацией.

## Стратегия Сканирования

Сканирование идёт от общего к частному, в три прохода:

### Проход 1 — Разведка

1. Получи общую картину: структура директорий, ключевые файлы, точки входа.
2. Читай файлы проекта — пойми стек, зависимости, назначение.
3. Найди точки входа — откуда стартует исполнение.
4. Сформулируй черновой `purpose` — одним абзацем, что это за система.

### Проход 2 — Карта сущностей и flows

1. Пройди по коду от точек входа вглубь. Отслеживай:
   - **Runtime flows** — цепочки действий, которые происходят при каждом событии/взаимодействии. Это кандидаты в modules.
   - **Статические сущности** — конфиги, модели данных, типы, API-контракты. Это кандидаты в definitions.
   - **Внутренняя механика** — хелперы, утилиты, координация между подсистемами. Это НЕ отдельные modules и НЕ отдельные definitions. Это детали реализации, которые упоминаются в body_markdown шагов или definitions, но не становятся самостоятельными единицами.
2. Группируй flows по зонам ответственности. Сигналы границы модуля:
   - Смена актора/исполнителя (пользователь → сервер → внешний API)
   - Смена режима обработки (синхронный → асинхронный)
   - Смена домена (auth → billing → notification)
3. Читай всё, что нужно для понимания flow. Тесты можно пропускать.

**Фильтр уровня абстракции.** При картировании задавай вопрос: "пользователь системы видит это действие или решение?" Если да — это шаг в module. Если нет (внутренняя координация, синхронизация, file locking) — это деталь реализации, но её ПОВЕДЕНИЕ может быть важным. Не делай отдельный module для координации, но опиши ключевые state-переходы и правила в definition или body_markdown шага, который запускает эту координацию.

**Что сохранять при абстрагировании:**
- Все состояния системы и переходы между ними (state machine)
- Различия между типами данных, если они влияют на логику (какие изменения считаются "содержательными", а какие "служебными")
- Условия, при которых поведение системы меняется (адаптивные параметры, проверки доступа, режимы работы)
- Побочные эффекты действий (изменение данных автоматически запускает другие процессы)

### Проход 3 — Детализация

1. Для каждого найденного flow разложи на шаги: что происходит, в каком порядке, где ветвления. Шаг = действие или решение, видимое на уровне системы.
2. Для каждого шага определи links — переходы к следующим шагам с условиями.
3. Для статических сущностей напиши definitions — что это, как работает, какие поля.
4. Свяжи шаги и definitions через `[[id]]` ссылки.
5. Перечитай результат и проверь: если в body_markdown шага есть имя функции или внутренней переменной — замени на описание того, что эта функция делает на уровне системы.

## Что Попадает В Модули

Модуль = повторяющийся runtime flow, видимый на уровне системы. Проверка: "этот flow выполняется при каждом использовании системы, и пользователь (или внешний актор) видит его результат?"

Примеры модулей:
- Обработка входящего HTTP-запроса в API gateway
- CI pipeline от коммита до деплоя
- Цикл "вопрос → обработка → ответ" в чат-боте
- Процесс оформления заказа в e-commerce

Что НЕ модуль:
- Внутренняя координация между подсистемами (sync, scheduling, state reconciliation) — это деталь реализации шага
- Setup/teardown/migration — одноразовые действия → definitions
- Регистрация компонентов, инициализация — инфраструктура → definitions
- Механика retry, fallback, circuit breaker — внутренняя механика → упоминание в body_markdown

## Что Попадает В Definitions

Definition = сущность предметной области, на которую ссылаются из flow.

Примеры definitions:
- Модели данных, форматы файлов, схемы БД
- Конфигурация, настройки, переменные окружения
- API-контракты (endpoints, форматы запросов/ответов)
- Внешние сервисы и интеграции
- Setup/migration/deploy — одноразовые действия
- CLI-команды, административные операции
- Координаторы и state machines — правила переходов между состояниями

Что НЕ definition:
- Внутренние абстракции кода (state objects, helpers, locks) — если пользователь не взаимодействует с этим напрямую, это не definition. Упомяни в body_markdown шага или существующей definition.

## Связность Модулей

Модули не должны быть изолированными островами. В реальной системе flows связаны: одно событие порождает другое, один модуль передаёт управление другому.

Правила:
- Если модуль A порождает событие, на которое реагирует модуль B — между ними должна быть кросс-модульная стрелка (link из шага A в шаг B).
- В большинстве систем есть один **центральный flow** — основной цикл обработки событий или взаимодействия с пользователем. Остальные модули подключаются к нему стрелками.
- Если все модули читаемы только по отдельности, но не складываются в общую картину — структура неправильная.
- **Порядок модулей в массиве `modules` = порядок чтения.** Первый модуль — центральный flow, с которого начинается понимание системы. Последующие — в порядке, в котором пользователь с ними сталкивается при использовании.

Проверка: убери все вспомогательные модули. Оставшийся центральный должен описывать основное поведение системы от входа до выхода. Вспомогательные модули связаны с ним через кросс-модульные стрелки.

Примеры связности — в `references/abstraction-guide.md`.

## Полнота Flow

Перед финализацией YAML проверь каждый модуль:

- **Все пользовательские действия покрыты?** Если пользователь может выполнить действие X, и оно меняет поведение системы — оно должно быть шагом или ветвлением. Если в flow есть "выход из состояния", должен быть и "вход в состояние".
- **Все исходы шагов видны?** Если шаг принимает решение с несколькими вариантами — каждый вариант должен быть стрелкой или описан в body_markdown.
- **Хуки и фоновые процессы — не шаги в чужих модулях.** Если система имеет фоновый процесс (хук, listener, watcher), который срабатывает на разные события — это definition, а не шаг в модуле обработки.
- **State-переходы координатора.** Если в коде есть координатор, который реагирует на изменения состояния по-разному (N сценариев) — все N сценариев должны быть описаны в definition координатора или в body_markdown шага, который вызывает координацию.
- **Нет дублирования definition ↔ flow.** Примеры — в `references/abstraction-guide.md`.

## Жёсткие Правила

- Не выдумывай логику, которой нет в коде. Описывай то, что видишь.
- Если flow непонятен — оставь грубый блок, не додумывай внутреннюю структуру.
- Если часть кода не удалось проанализировать — скажи об этом, не скрывай.
- Условия на стрелках — человеческим языком, не кодом. Примеры перевода — в `references/abstraction-guide.md`.
- ID модулей и шагов — семантические, отражают роль, а не имя файла или класса.

## Порядок Работы

1. Уточни у пользователя, если нужно: что сканировать, есть ли фокус.
2. Прочитай `references/abstraction-guide.md` — примеры уровня абстракции и антипаттерны.

## Формат Выхода

Ровно тот же формат, что у yaml-author:

```yaml
purpose: |
  Что делает система, для кого, ключевые принципы.

definitions:
  - id: название сущности
    body_markdown: |
      Описание. Ссылки через [[другой id]].

modules:
  - id: semantic-module-id
    purpose: Зачем этот модуль существует.
    steps:
      - id: semantic-step-id
        body_markdown: |
          Логика шага. Ссылки через [[id]].
        links:
          - to: another-step-id
            condition: Условие человеческим языком.
```

Канонический пример формата — в `.claude/skills/yaml-author/references/example.md`.
Пример входа и результата: `references/example-input.md` → `references/example-output.md`. Прочитай оба, чтобы понять какой уровень абстракции и стиль ожидается.
Частые ошибки формата — в `.claude/skills/yaml-author/references/common-mistakes.md`.

## Рекомендации По Стилю

- Пиши `body_markdown` так, чтобы человек, впервые открывший YAML, понял что происходит. Не используй внутреннюю терминологию системы без пояснения. Если формулировка понятна только тому, кто уже знает систему — перепиши проще.
- `purpose` модуля — зачем flow существует, а не пересказ шагов.
- `body_markdown` — списки, `[[ссылки]]`, блоки кода для конкретных команд/API.
- Условия на links — короткие фразы на языке предметной области.
- Для недоопределённых частей — грубый блок с честным описанием.
- Не добавляй заголовки (`#`, `##`) внутри `body_markdown`.
