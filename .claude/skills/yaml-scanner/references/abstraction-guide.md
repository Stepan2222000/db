# Руководство по уровню абстракции

Scanner видит код — функции, классы, переменные. YAML описывает систему на языке предметной области. Это руководство показывает как переводить с одного на другое.

## Код → Система: примеры перевода

### Имена функций → действия

Плохо (уровень кода):
- `processPaymentWithRetry(order, 3)` обрабатывает платёж с тремя попытками
- `eventBus.emit("user.updated", payload)` публикует событие обновления
- Под `withTransaction` обновляет баланс и создаёт запись

Хорошо (уровень системы):
- Списать средства с баланса пользователя
- Уведомить связанные сервисы об изменении профиля
- Обновить баланс и записать транзакцию

### Имена полей → описания

Плохо:
- `ping_pending: true` → планируем retry и ignore_timeout
- снятие `awaiting_user_after_ignore` открывает новый цикл
- `last_ping_at` не обновляется при retry

Хорошо:
- пинг отправлен → планируем повтор и таймаут игнора
- пользователь написал после игнора → возобновляем обычный цикл
- время первого пинга не меняется при повторе

### Условия на стрелках

Плохо:
- `if (status === 401)` → redirect
- `mode === "sleeping"` → skip
- `payload.type === "auto"` → process

Хорошо:
- не авторизован
- пользователь спит
- автоматическая обработка

### Правило

Если читатель, не знакомый с кодом, не понимает что означает термин — замени на описание действия на языке предметной области.

## Дублирование: definition ↔ flow

Если логика описана в definition — шаг flow ссылается через `[[id]]`, а не пересказывает.

### Плохо — дублирование

```yaml
definitions:
  - id: правила валидации
    body_markdown: |
      Заказ проходит три проверки: наличие товара,
      лимит суммы, статус клиента. Если любая не пройдена —
      заказ отклоняется с причиной.

modules:
  - id: order-flow
    steps:
      - id: validate-order
        body_markdown: |
          Проверить заказ. Три проверки: наличие товара,
          лимит суммы, статус клиента. Если любая
          не пройдена — отклонить с причиной.
          # ^^^ То же самое что в definition!
```

### Хорошо — ссылка

```yaml
modules:
  - id: order-flow
    steps:
      - id: validate-order
        body_markdown: |
          Проверить заказ по [[правила валидации]].
```

## Связность модулей

Модули не должны быть изолированными островами. Если в системе есть несколько runtime flow — они связаны через кросс-модульные стрелки или общие шаги.

### Плохо — острова

```yaml
modules:
  - id: order-processing
    steps:
      - id: process-order
        body_markdown: Обработать заказ...
        # Нет стрелок наружу

  - id: notification-sending
    steps:
      - id: send-notification
        body_markdown: Отправить уведомление...
        # Нет стрелок внутрь — откуда берутся уведомления?
```

### Хорошо — связанный граф

```yaml
modules:
  - id: order-processing
    steps:
      - id: process-order
        body_markdown: Обработать заказ...
        links:
          - to: send-notification
            condition: заказ подтверждён

  - id: notification-sending
    steps:
      - id: send-notification
        body_markdown: Отправить уведомление...
```

## Центральный flow

В большинстве систем есть один главный flow — основной цикл взаимодействия с пользователем или обработки событий. Остальные модули подключаются к нему.

Если в YAML нет модуля, который отвечает за "что происходит когда приходит основное событие системы" — структура неполная.

Проверка: убери все вспомогательные модули. Оставшийся должен описывать основное поведение системы от входа до выхода.
