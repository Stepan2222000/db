# Частые Ошибки

## 1. Модуль без flow

**Ошибка:** модуль, в котором шаги не связаны стрелками — просто набор карточек.

```yaml
# Плохо — модуль описывает структуру, а не flow
modules:
  - id: data-model
    purpose: Хранение состояния в файлах.
    steps:
      - id: status-file
        body_markdown: |
          Файл status.md — текущий snapshot...
      - id: daily-files
        body_markdown: |
          Дневные файлы YYYY-MM-DD.md...
      - id: config
        body_markdown: |
          Конфиг в openclaw.json...
```

**Почему плохо:** на canvas это три несвязанных карточки в рамке. Нет стрелок, нет порядка, нет ветвлений — нет визуальной ценности. Читатель не понимает, что здесь происходит, потому что ничего не происходит. Это справочная информация, а не flow.

**Правильно:** вынести в definitions.

```yaml
definitions:
  - id: status.md
    body_markdown: |
      Файл `memory/status.md` — текущий snapshot...
  - id: дневной файл
    body_markdown: |
      Дневные файлы `memory/YYYY-MM-DD.md`...
  - id: конфиг плагина
    body_markdown: |
      Блок в `openclaw.json`...
```

**Проверка:** если в модуле нет ни одного link между шагами — скорее всего это не модуль.

## 2. Таблицы в definitions

**Ошибка:** markdown-таблица в body_markdown definition.

```yaml
# Плохо — таблица не рендерится в модалке
- id: конфиг
  body_markdown: |
    | Поле | Дефолт | Назначение |
    |------|--------|-----------|
    | chatId | — | ID чата |
    | timeout | 30 | Таймаут |
```

**Почему плохо:** модалка definitions рендерит markdown, но таблицы отображаются как сырой текст с разделителями `|`. Нечитаемо.

**Правильно:** списки.

```yaml
- id: конфиг
  body_markdown: |
    Поля:
    - `chatId` (обязательно) — ID чата
    - `timeout` (`30`) — таймаут в секундах
```

## 3. Технический язык в условиях на стрелках

**Ошибка:** condition содержит названия полей, операторы сравнения, код.

```yaml
# Плохо
links:
  - to: skip-action
    condition: mode = sleeping
  - to: check-timeout
    condition: ping_pending = true && elapsed > threshold
```

**Почему плохо:** граф должен читаться как диаграмма процесса — без знания кода. Стрелки отвечают на вопрос "когда идём сюда?" на языке предметной области.

**Правильно:**

```yaml
links:
  - to: skip-action
    condition: пользователь спит
  - to: check-timeout
    condition: пинг ждёт ответа
```

## 4. Дублирование между шагами и definitions

**Ошибка:** body_markdown шага пересказывает то, что уже есть в definition.

```yaml
# Плохо — шаг повторяет содержимое definition
- id: update-status
  body_markdown: |
    Вызвать tracker_update — partial update status.md,
    read-modify-write атомарно, auto updated_at.
    Обновить поле active...
```

**Правильно:** шаг описывает действие, definition — инструмент.

```yaml
- id: update-status
  body_markdown: |
    Обновить активность через [[tracker_update]],
    записать событие через [[tracker_log]].
```

Читатель понимает ЧТО происходит. Если хочет узнать КАК работает tracker_update — кликает на `[[tracker_update]]`.

## 5. Одноразовые действия как модуль

**Ошибка:** модуль для действий, которые выполняются один раз при развёртывании, а не при каждом использовании системы.

```yaml
# Плохо — деплой БД, миграция, bootstrap — не runtime flow
modules:
  - id: database-setup
    purpose: Подготовить базу данных для работы сервиса.
    steps:
      - id: run-migrations
        body_markdown: |
          Запустить миграции, создать таблицы...
        links:
          - to: seed-data
      - id: seed-data
        body_markdown: |
          Заполнить начальные справочники...
```

**Почему плохо:** модули описывают повторяющееся поведение — то, что система делает при каждом запросе, событии, взаимодействии. Подготовка окружения, деплой, миграции — это предпосылки для работы, а не сама работа. На графе такой модуль визуально конкурирует с runtime-логикой, хотя выполняется один раз и больше не повторяется.

**Проверка:** "этот flow повторяется при каждом использовании системы?" Если нет — это не модуль.

**Правильно:** вынести в definitions.

```yaml
definitions:
  - id: подготовка базы данных
    body_markdown: |
      Перед первым запуском:
      - Запустить миграции (`db migrate`)
      - Заполнить справочники (`db seed`)
      - Проверить подключение
```
