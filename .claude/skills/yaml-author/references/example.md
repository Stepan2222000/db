# Канонический Пример

Этот пример показывает:

- трёхслойную структуру: purpose → definitions → modules
- definitions для технических сущностей с `[[id]]` ссылками
- modules только для runtime flow (действия, решения, ветвления)
- грубый модуль для недоопределённого пути
- условия на стрелках человеческим языком
- списки вместо таблиц в definitions

```yaml
purpose: |
  Система обработки входящих запросов. Принимает запрос,
  обогащает контекст, генерирует ответ. High-risk запросы
  уходят на ручную проверку.

definitions:
  - id: входящий запрос
    body_markdown: |
      JSON payload от внешнего API. Обязательные поля:
      - `tenant_id` — идентификатор арендатора
      - `type` — тип запроса (`auto` / `manual` / `priority`)
      - `payload` — тело запроса (свободная структура)

      Валидация: [[валидатор полей]] проверяет обязательные поля
      и формат перед маршрутизацией.

  - id: валидатор полей
    body_markdown: |
      Проверяет обязательные поля [[входящий запрос]]:
      - Наличие `tenant_id`, `type`, `payload`
      - Формат `type` — одно из допустимых значений
      - Размер `payload` не превышает лимит

      Невалидный запрос → немедленное отклонение с причиной.

  - id: контекст обогащения
    body_markdown: |
      Дополнительные данные, собранные перед генерацией ответа:
      - Tenant context (настройки арендатора)
      - Feature flags (включённые функции)
      - Compliance tags (ограничения по юрисдикции)
      - Formatting rules (формат ответа)

      Собирается автоматически на основе `tenant_id`
      из [[входящий запрос]].

  - id: политика маршрутизации
    body_markdown: |
      Правила, определяющие путь обработки запроса:
      - `type: auto` → автоматическое обогащение
      - `type: manual` → ручная проверка
      - `type: priority` → автоматическое, но с уведомлением

      Правила хранятся в конфиге системы и подставляются при старте.

modules:
  - id: request-intake
    purpose: Принять запрос, проверить и направить на нужный путь обработки.
    steps:
      - id: read-request
        body_markdown: |
          Прочитать [[входящий запрос]] и нормализовать
          payload во внутренний формат.
        links:
          - to: validate-fields
            condition: запрос прочитан

      - id: validate-fields
        body_markdown: |
          Проверить обязательные поля через [[валидатор полей]].
        links:
          - to: route-request
            condition: запрос валиден
          - to: reject-request
            condition: запрос невалиден

      - id: route-request
        body_markdown: |
          Определить путь обработки по [[политика маршрутизации]].
        links:
          - to: enrich-overview
            condition: автоматическая обработка
          - to: manual-review-overview
            condition: требуется ручная проверка

      - id: reject-request
        body_markdown: |
          Остановить обработку, вернуть пользователю причину отклонения.

  - id: enrichment
    purpose: Собрать дополнительный контекст перед генерацией ответа.
    steps:
      - id: enrich-overview
        body_markdown: |
          Собрать [[контекст обогащения]] на основе данных
          из [[входящий запрос]].

          Точная последовательность сбора ещё не согласована —
          оставлен грубый блок.
        links:
          - to: finalize-response
            condition: обогащение завершено

  - id: response-finalization
    purpose: Собрать финальный ответ из подготовленного контекста.
    steps:
      - id: finalize-response
        body_markdown: |
          Сформировать ответ из [[входящий запрос]]
          и [[контекст обогащения]].

  - id: manual-review
    purpose: Обработать запросы, которые не могут быть обработаны автоматически.
    steps:
      - id: manual-review-overview
        body_markdown: |
          Ветка ручной проверки. Внутренний workflow
          ещё не согласован — грубый блок.
```
