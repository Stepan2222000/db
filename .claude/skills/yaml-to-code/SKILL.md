## Входы

- Путь к YAML-файлу (обязательно).
- Целевая директория для проекта (обязательно).
- Стек/язык/фреймворк — если пользователь указал.

## Цель

Превратить YAML-спецификацию в полностью рабочий проект. YAML — это контракт: каждый module, step и definition — обещание поведения. Код реализует именно это, не больше и не меньше.

Ты — автономный разработчик. На каждом шаге работаешь столько, сколько нужно. Тестируешь все реальные случаи. Но лишнего кода не пишешь.

## Как Читать YAML

- `purpose` — что система делает, для кого, ключевые принципы.
- `definitions` — технические сущности. `[[id]]` — ссылка на definition.
- `modules` → `steps` → `links` — граф flow. Steps — действия, links — переходы.

Пример YAML-документа — в `references/example-yaml.md`. Прочитай его перед первым анализом.

## Два режима работы

### Режим 1: Анализ и план

**Срабатывает когда `implementation-plan.md` не существует в директории проекта.**

В этом режиме ты НЕ пишешь код. Только анализируешь и планируешь.

1. Прочитай весь YAML. Извлеки: что система делает, какие данные, какие flow, какие зависимости.

2. Проведи глубокий ресерч в интернете по стеку и зависимостям. Прочитай ссылки на документацию из definitions.

3. Проверь окружение: что нужно для разработки и тестирования (платформы, runtime, CLI, SDK). Проверь что установлено. Попробуй запустить. Если чего-то нет — запиши как блокер.

4. Создай `implementation-plan.md` в директории проекта:
   - Краткое понимание системы
   - Список блокеров окружения (что не установлено, что нужно)
   - Разбивка на куски (маленькие шаги реализации, по порядку)
   - Для каждого куска: что делается, как тестируется, минимальное время работы
   - E2E сценарии для финальной проверки
   - Технические инструменты тестирования
   - Вопросы пользователю

5. После каждого ответа пользователя — глубоко анализируй заново. Ищи в интернете по новой информации. Задавай следующие вопросы. Обновляй план.

6. **СТОП.** Не пиши код. Жди пока пользователь скажет "приступай" или вызовет скилл снова.

### Режим 2: Реализация одного куска

**Срабатывает когда `implementation-plan.md` уже существует.**

1. Прочитай `implementation-plan.md`. Найди следующий невыполненный кусок.

2. Реализуй только этот кусок:
   - Напиши код
   - Собери
   - Протестируй все реальные сценарии для этого куска
   - Проверь что предыдущие куски не сломались
   - Коммить

3. Обнови `implementation-plan.md`: отметь кусок как выполненный, запиши что протестировано, запиши вопросы если появились.

4. Если есть вопросы — задай. Если нет — напиши что сделано и что будет дальше.

5. **СТОП.** Жди следующего вызова для следующего куска.

Когда все куски выполнены — пройди E2E сценарии из плана. Сообщи итог.

## Жёсткие Правила

- Не выдумывай поведение, которого нет в YAML. Не пропускай поведение, которое есть.
- Не занимайся overengineering. Что на практике не встречается — не обрабатывай и не тестируй. Лишнего кода быть не должно.
- Ресерч — глубокий. Изучи документацию досконально до написания кода.
- Тестирование — все реальные случаи. Но не выдуманные.
- Если YAML содержит `[[ссылку]]` — найди definition и реализуй описанное.
- Если в definition есть ссылки на документацию — прочитай их.
- Если не знаешь как реализовать или протестировать — ищи в интернете.
- Если что-то невозможно (нужен внешний сервис, API-ключ) — заглушка с TODO.
- Коммить после каждого куска.
- В режиме 1 — НЕ ПИСАТЬ КОД. Только анализ и план.
- В режиме 2 — ТОЛЬКО ОДИН КУСОК за вызов.
