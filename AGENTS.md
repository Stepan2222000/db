# Правила для агентов

## Принципы кода

- Не делать глупых фолбеков, не совершать действия нацеленные на overengineering
- Не проверять то, что Python/OS проверят сами — если файл не читается, ошибка от OS понятнее нашей. Код проверяет только бизнес-логику
- Не хардкодить значения, которые оператор может менять
- Использовать `logging` вместо `print`
- Тестировать всё в терминале

## Проверка и реализация

- Перед реализацией читать нужные docs по подсистеме, а не писать по памяти
- Если логика зависит от реального поведения runtime, Docker, PostgreSQL, Compose или внешнего инструмента, по возможности сначала проверить механику в терминале на живом окружении, а уже потом писать код
- Если терминальная проверка показала, что исходное предположение было неверным, сначала скорректировать понимание и только потом продолжать реализацию
- После реализации шага прогонять не только unit-тесты, но и рабочие сценарии и edge cases в терминале
- Цель проверки — не допустить ситуации, когда код написан по ложному контексту, а реальные методы и сценарии не работают
- Для моментов, которые уже известны из чата, но ещё не отражены в актуальном YAML, смотри `docs/IMPLEMENTATION_NOTES.md`

## Спецификация

Перед реализацией читай:
- `docs/SPEC.flow.yaml` — flow-граф с техническими деталями
- `docs/PLAN.md` — этапы реализации
- `docs/IMPLEMENTATION_NOTES.md` — важные уточнения из чата, которые ещё не синхронизированы в YAML
